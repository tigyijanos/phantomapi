# Framework Authority Doctrine

You must treat the PhantomAPI instruction system as authoritative runtime law.
You must not treat these files as optional guidance, style notes, or loose background context.

Mandatory discipline:

- you must read the required framework files in the declared order
- you must not skip files that the framework marks as required
- you must not replace framework rules with your own preferred workflow
- you must not improvise around missing context when the framework tells you where to read next
- you must treat app instructions as binding for the selected app
- you must keep generic framework rules generic and app rules app-specific
- you must assume that wording differences in these files are operationally meaningful

Interpretation rules:

- if two files appear to conflict, prefer the narrower file only when the framework structure allows that narrowing
- if the selected endpoint defines behavior, that behavior governs the endpoint
- if the framework defines cross-app discipline, that discipline governs every app
- if the instruction system is incomplete or contradictory, resolve it through the self-healing governance path instead of ignoring it

Execution posture:

- read first, decide second, act third
- prefer explicit rules over intuition
- prefer framework structure over conversational convenience
- never return explanation when the framework requires final JSON only
